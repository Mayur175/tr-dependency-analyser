# ADR-001: Dual ABAP source tree (Standard ABAP + ABAP for Cloud Development)

- **Status:** Accepted
- **Date:** 2026-06-19 (extracted from existing docs on 2026-06-20)
- **Deciders:** @Mayur175

## Context

The TR Dependency Analyser must run on:
- S/4HANA on-prem / Private Cloud — Standard ABAP, classic table SELECTs
  permitted (`E070`, `E071`, `SEOMETAREL`, `DD03L`, `DD04L`, `TFDIR`).
- S/4HANA Public Cloud / BTP ABAP Environment — ABAP for Cloud Development,
  strict allow-list, classic-table SELECTs blocked at compile time.

A single source tree cannot satisfy both, because the cloud allow-list is
enforced at activation and would reject the same code that compiles cleanly
on-prem.

## Decision

Maintain **two parallel ABAP source trees** with the **same public API
surface** (constructor signature, `to_json()`, `to_csv()`, `persist_result()`,
HTTP endpoint shape):

- `abap/` — Standard ABAP backend.
- `abap_cloud/` — ABAP for Cloud Development backend.

Customers install **one or the other**, never both, depending on their
landscape. The Eclipse plugin works against either backend without code
changes because the JSON contract is identical.

## Consequences

- **Positive:** clean separation of concerns; either backend can be
  installed with abapGit on the matching landscape; no runtime feature
  detection needed in the inventory step.
- **Negative:** double maintenance — bug fixes and feature additions must
  be ported between trees. The trees can drift if discipline lapses.
- **Neutral:** README must clearly direct users to the correct tree
  (this is now done in the install table).

## Verification

ATC clean on each tree against its target system. Cloud tree must produce
zero "use of element X is not permitted" errors on a Public Cloud tenant.
