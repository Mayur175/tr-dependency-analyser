# ADR-002: Use classic CTS / DDIC tables as the universal data path

- **Status:** Accepted
- **Date:** 2026-06-19 (extracted from existing docs on 2026-06-20)
- **Deciders:** @Mayur175

## Context

XCO (`xco_cp_cts`, `xco_cp_oo`, `xco_cp_abap_dictionary`) is the
SAP-recommended cloud API surface, but its availability varies:

| System | XCO present? |
|---|---|
| BTP ABAP Environment / S/4HANA Public Cloud | Yes |
| S/4HANA Private Cloud / on-prem 2021+ | Yes (most APIs back-ported) |
| S/4HANA on-prem 1909 / 2020 | Partial; verify per SP |
| NetWeaver / S/4HANA <= 1809 | No |
| Classic ECC | No |

A pure-XCO implementation would not run on a large fraction of the
target landscape. Conversely, classic-table SELECTs work on **every**
SAP NetWeaver / S/4HANA release.

## Decision

The current implementation uses **only classic CTS / DDIC tables** for
inventory and dependency extraction. The XCO path is documented as a
preferred-when-available future addition behind a runtime feature-detect
flag (`cl_abap_classdescr=>describe_by_name( 'XCO_CP_CTS' ) IS BOUND`),
but is not yet implemented.

For the cloud tree, the table reads are bypassed entirely — the cloud
variant uses `xco_cp_cts` directly because the cloud allow-list forbids
the table SELECTs. The cloud tree therefore intentionally does NOT have
a classic-table fallback.

## Consequences

- **Positive:** Standard ABAP tree runs on every release; no
  feature-detection complexity in v1; tables (`E070`/`E071` since R/3,
  `SEOMETAREL` since 4.6C, `DD03L`/`DD04L` since R/3) are stable across
  releases.
- **Negative:** Cloud tree and Standard ABAP tree do not share extractor
  code — see ADR-001 for the implications.
- **Neutral:** XCO can be added later behind feature detection without
  breaking either current path.

## Verification

Backend functional on a system without XCO (e.g. on-prem 1809) is the
acceptance test for this decision.
