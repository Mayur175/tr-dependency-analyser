# ADR-003: ICF handler ships with AUTHORITY-CHECK disabled by default

- **Status:** Accepted, **but operators MUST flip before production use**
- **Date:** 2026-06-19 (extracted from existing source comment on 2026-06-20)
- **Deciders:** @Mayur175

## Context

The ICF handler at \`/sap/bc/zgcts/analyze\` reads transport-request
contents — formally a privileged operation requiring \`S_TRANSPRT\`
(TTYPE=\`CUST\`, ACTVT=\`03 Display\`).

Pilot users on personal / dev tenants typically do **not** have basic
authorisation to grant themselves new auth objects, and waiting for
Basis support delays adoption. The original goal was to lower the
install friction so the tool could be tried in an afternoon.

## Decision

Ship with the constant \`c_enforce_auth\` (or equivalent flag) set to
\`abap_false\`, which **skips AUTHORITY-CHECK**. The handler emits an
\`X-Auth-Bypass: yes\` HTTP response header on every request so that
downstream monitoring / SIEM can detect the open posture.

The handler source carries a clearly-marked \`>>> WARNING - SECURITY
CRITICAL <<<\` comment block (see
[abap/src/zgcts_analyze_handler.clas.abap](../../abap/src/zgcts_analyze_handler.clas.abap))
explaining the trade-off and listing the production-flip steps.

## Consequences

- **Positive:** zero-friction install on pilot/sandbox tenants.
- **Negative:** **on a shared development, QA, or production tenant
  this exposes every TR's contents to every authenticated SAP user with
  HTTP access**. This is real risk; documentation alone does not enforce
  the flip.
- **Neutral:** the \`X-Auth-Bypass\` header makes the open posture
  detectable in logs.

## Verification

Operations team must observe absence of the \`X-Auth-Bypass: yes\` header
in production HTTP traces. Any presence of the header on a non-sandbox
system is an incident.

## Re-evaluation

This decision should be re-evaluated when the install experience can
support a "production-safe" mode by default — e.g. a configuration table
that the installer fills in during abapGit pull.
